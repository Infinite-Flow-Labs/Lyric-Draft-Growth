My old mistake: I judged AI tools by features, not by friction.  
A tool could summarize PDFs, build slides, and “automate workflows,” but still cost me more time after day 3.

So I switched to a simple execution rubric. If a tool can’t pass this in one test session, I don’t adopt it.

I run the same 5 checks, in this exact order:

I start a 30-minute timer and use one real task I already do weekly (not a fake demo task).  
Example: turn 6 messy meeting notes into a publish-ready update.

I score setup friction first: account creation, permissions, and first usable output.  
If setup takes more than 8 minutes, it gets a penalty.  
If I need to watch a tutorial before first output, I usually drop it.

Then I test output quality under constraints:  
“Give me a 120-word update, 3 bullet decisions, and 2 action owners.”  
If I spend more than 5 minutes rewriting structure, quality score is low.

Next is reliability: I run the same prompt 3 times with minor input changes.  
If format consistency is below 2/3, it fails for operational use.

Then I check handoff cost: can I export to my real stack (Docs, Notion, Slack, Calendar) in under 2 clicks?  
If copy-paste cleanup takes over 3 minutes, it’s not “saving time.”

Finally, I calculate a blunt metric:  
**(minutes saved per week) - (weekly maintenance minutes).**  
If net gain is under 20 minutes/week after 7 days, I archive it.

This rubric cut my active AI tools from 11 to 4.  
Result: weekly workflow interruptions dropped from 19 to 7, and my task completion consistency (planned vs shipped) improved from 61% to 84% in two weeks.

What I keep: fast setup, stable format, low handoff cost.  
What I delete: “powerful” tools that need constant babysitting.

This works well when you run repeatable tasks (content, admin, planning, reporting) and care about steady output.  
If your work is mostly novel research, experimental design, or high-stakes analysis, this filter can be too strict early on—you may reject tools that need longer ramp time but eventually become valuable.

我会按三步推进：先把输入和验收写清，再跑最小闭环，最后只保留有效动作并沉淀成模板。
这套方法更适合高频、可验收的重复任务；一次性探索型问题不建议硬套。
