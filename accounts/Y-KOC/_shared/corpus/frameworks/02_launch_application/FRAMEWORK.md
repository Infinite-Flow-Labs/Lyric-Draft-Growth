# 02_launch_application

## 父框架名称

发布后落地型

旧称：发布应用型（Launch -> How I Use）

## 定义

不是单纯转述“某个新东西发布了”，而是：

- 把发布信号翻译成读者能立刻理解的价值变化
- 把产品、模型、功能、文档的抽象能力压成可执行的使用路径
- 把“知道消息”推进到“知道该不该用、怎么开始、适合什么场景”

这个父框架的价值不在于抢首发，而在于**缩短发布消息和真实采用之间的距离**。

## 核心任务

这个父框架通常要完成 4 件事：

1. 说清楚这次发布的对象到底是什么
2. 解释它解决了什么旧痛点，或创造了什么新能力
3. 告诉读者应该怎样开始使用、测试或接入
4. 让读者形成采用判断，而不是停留在“知道了”

## 共享规则

无论下面走哪种结构参数包，`02_launch_application` 都必须满足这些规则：

1. **必须存在明确的发布触发点**
   没有新品、更新、开源、上新、套餐、文档发布等信号，就不属于这个父框架。

2. **必须把“功能”翻译成“收益或动作”**
   不能只罗列 feature，必须回答“这对你有什么变化”。

3. **必须降低采用摩擦**
   至少要降低其中一种摩擦：理解成本、安装成本、判断成本、上手成本。

4. **必须有真实证明**
   可以是步骤、案例、测试、截图、真实任务、配置、价格、体验对比，但不能只剩宣传语。

5. **必须保留采用边界**
   真正可复用的发布应用文，不会只告诉你“能用”，还会告诉你“适合什么，不适合什么”。

## 共享总骨架

`02_launch_application` 的隐式骨架通常是：

1. `Release signal / what's new`
2. `Why this matters / old friction or new leverage`
3. `What it actually is`
4. `How to start / where it fits`
5. `Proof through steps, scenarios, or tests`
6. `Adoption judgment / CTA`

这 6 段是后台逻辑，不要求未来生成时把它们显式写成标题。

## 重要原则：框架是隐式采用逻辑，不是显式教程模板

后续生成时，绝不能把这个父框架写成统一可见模板：

- 先说发布
- 再说功能
- 再说怎么用
- 最后放链接

那样只会像模板化产品文。

### 正确用法

把这套骨架当成“采用判断控制逻辑”，而不是“文章排版模板”。

允许：

- 先从旧方案的痛点开局，再引出这次发布
- 先讲一个真实测试，再回头解释新功能
- 先给一个使用结论，再拆适用边界
- 用步骤、案例、场景、截图说明、价格套餐、配置片段承载骨架

不允许：

- 每篇都写成“XX 发布了，下面告诉你怎么用”
- 每篇都用完全对称的 1/2/3/4 小标题
- 每篇都只会列 feature，不讲采用判断
- 每篇都把真实使用场景磨平成新闻稿

## 结构参数轴

`02_launch_application` 不应继续平铺成越来越多框架。统一只在这些轴上变化：

- `reader_state`
  - `heard_the_news_but_not_the_use`
  - `wants_the_fastest_onboarding`
  - `needs_adoption_judgment`

- `release_object`
  - `feature`
  - `app_or_product`
  - `model`
  - `system`
  - `guide_or_resource`

- `proof_mode`
  - `quickstart_steps`
  - `hands_on_test`
  - `scenario_demo`
  - `operating_principles`

- `stance`
  - `instructional`
  - `corrective`
  - `enthusiastic`
  - `craftsmanship`

## 3 组稳定结构参数包

### A. feature_playbook

#### 适用

- 新 feature / 新能力 / 新配置方式发布后
- 读者不是来围观消息，而是想知道“这东西到底该怎么用，什么时候该用，不该怎么用”

#### 对应样本

- How to run subagents in Codex
- Claude 给了你 1M Token 的上下文，但别真的全塞满
- You Should Be Using Subagents in Codex!

#### 稳定段落流

1. `Release signal`
2. `What changed in plain language`
3. `How to use or operate it`
4. `Best use cases`
5. `Boundaries / mistakes / when not to use`
6. `Quick action`

#### 常见钩子

- `X just dropped...`
- `Anthropic 官宣...但别真的全塞满`
- `You should be using...`
- `这绝对是好事，但...`

#### 证据组合

- 机制解释
- prompt / config / TOML 片段
- best use cases
- 反例或误用场景
- 快速上手动作

#### 语言动作

- 很快从“发布了什么”切到“你该怎么用”
- 直接定义概念，少绕
- 常见 `what it is / how to use / when not to use`
- 允许轻微纠偏口吻，但仍然以采用建议为终点

#### 改写禁区

- 只像新闻转述，不给操作路径
- 只有步骤，没有采用判断
- 只有夸赞，没有边界
- 写成厚重官方文档，失去转述感

#### 改写公式

`发布信号 -> 白话翻译 -> 使用方式 -> 适用场景 -> 边界或误区 -> 立即动作`

### B. pain_to_adoption

#### 适用

- 旧方案明显太难、太慢、太贵、太笨
- 新产品 / 新系统 / 新模型发布后，文章要做的是把“老痛点”转换成“新采用路径”

#### 对应样本

- 90%的亚马逊选品还不如算命，AI选品系统发布！！
- 龙虾专属模型来了，给你的虾换个好脑子吧
- 这应该使用 OpenClaw 最简单的方案了，点击安装就能用

#### 稳定段落流

1. `Current pain or broken baseline`
2. `Why old way fails`
3. `New release as the practical answer`
4. `Walkthrough / setup / operating path`
5. `Real task proof`
6. `Adoption close`

#### 常见钩子

- `说实话，用命令行...确实太费劲了`
- `90%的选品决策，本质上跟扔骰子没区别`
- `真正的问题是...`
- `你的虾，值得一个好脑子`

#### 证据组合

- 痛点场景
- 对比旧方法
- 安装或接入步骤
- 仪表盘 / 工作流 / 测试过程
- 价格、套餐、注册、领取、实际任务结果

#### 语言动作

- 先把旧方法打穿，再把新东西抬出来
- 叙述里会带明显的“终于能用了”“这次顺滑了”的采用感
- 中文样本更口语、更高对比，常用夸张和比喻压记忆点
- 既能卖点，也会给具体操作

#### 改写禁区

- 只剩夸张，不给真实采用路径
- 只给步骤，不先建立旧痛点
- 全文像广告，不像经验转译
- 演示能力很多，但没有“这为什么比旧方案更好”

#### 改写公式

`旧痛点 -> 旧路不通 -> 新发布是答案 -> 怎么接入或操作 -> 真实任务证明 -> 采用动作`

### C. release_showcase

#### 适用

- 开源工具、模型、文档、课程、资源库发布
- 读者首先需要被说服“这东西值得试”，而不是先看完整安装说明

#### 对应样本

- 清华团队开源 AI 课堂，人人都能学起来
- 《OpenClaw 从入门到精通指南》正式发布，开源免费！

#### 稳定段落流

1. `Release announcement`
2. `One-sentence positioning`
3. `What's inside`
4. `Concrete scenarios or examples`
5. `Why this matters / who it helps`
6. `Link or entry point`

#### 常见钩子

- `前两天...开源了个好东西`
- `今天，我们打磨了很久的...终于正式和大家见面了`
- `简单说，它是...`
- `更有意思的是...`

#### 证据组合

- 一句话定位
- 内含模块 / 内容清单
- 典型使用场景
- 亲自实践或制作过程
- 开源地址 / 入口链接

#### 语言动作

- 先把门槛降下来：先告诉读者“这到底是什么”
- 很依赖场景切换和“比如”
- 情绪可以偏热情，但要有实操感
- 常见“我们做了很久 / 我体验过 / 你现在就能试”

#### 改写禁区

- 一上来就抽象夸产品
- 只有 feature list，没有“怎么想象它的价值”
- 全是场景，没有对象定位
- 写成空泛发布通稿

#### 改写公式

`发布 -> 白话定位 -> 内容或能力展开 -> 场景证明 -> 价值判断 -> 入口链接`

## 关键收束：不要把“纠偏型发布文”单独拆成新框架

像 `Claude 1M context` 这种样本，虽然口吻更纠偏，但它本质上仍然属于 `feature_playbook`：

- 触发点仍然是发布
- 主体任务仍然是采用判断
- 只是 `stance = corrective`

也就是说，它是父框架下的口吻参数，不是新的一级框架。

## 改写时的双层控制

后续生成时，必须同时遵守两层控制：

1. **隐式采用结构层**
   确保文章不是散的，知道从发布走向采用。

2. **表层表达层**
   学习真实样本的语气、段落长度、切场景方式、说明动作、收尾力度。

如果只有第一层，会写得像教程模板。
如果只有第二层，会写得像热闹的产品帖，无法稳定改写。

## 语言风格学习原则

`02_launch_application` 学的不是“把发布消息改写成统一格式”，而是：

- 什么情况下应该直接解释
- 什么情况下应该先建痛点
- 什么情况下应该用场景证明能力
- 什么情况下应该加边界和误区
- 什么地方应该保留转述感，而不是写成官方说明书

## 抗 AI 味要求

后续生成时，尤其要避免：

- 每篇都以“下面我将介绍”开头
- 每篇都用一模一样的教程结构
- 标题很炸，正文像需求文档
- 段落过于对称，像产品经理列表
- 只会说“这很强大”“效率提升明显”，没有具体动作和证据
