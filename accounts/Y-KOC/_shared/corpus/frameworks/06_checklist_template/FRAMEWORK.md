# 06_checklist_template

## 父框架名称

能力交付型技术分享

旧称：技术分享型（工作流 / Skill / 提示词）

## 定义

把一个复杂能力、工具、Skill、系统、流程或方法，压缩成读者能够理解、能够照做、能够迁移的执行路径。

这个父框架的重点不是“介绍技术本身”，而是：

- 让读者知道为什么值得用
- 让读者知道怎么开始或怎么增强
- 让读者知道真正落地时要注意什么

## 核心任务

这个父框架要完成的不是“讲清楚全部知识”，而是完成下面 3 件事：

1. 把复杂能力翻译成具体收益
2. 把抽象方法压缩成可执行步骤或结构
3. 替读者降低试错成本

## 共享规则

无论下面走哪种结构参数包，`06_checklist_template` 都必须满足这些共享规则：

1. **必须先交代读者收益**
不能一上来堆功能点，必须先回答“读者为什么要继续看”。

2. **必须给出执行路径**
至少要让读者知道该怎么开始，而不是只停留在概念或观点。

3. **必须用具体证据支撑**
可以是步骤、命令、场景、对比、角色分工、项目结果、仓库链接，但不能只有抽象描述。

4. **必须降低读者的不确定性**
可以通过准备清单、步骤总览、前后对比、FAQ、坑点、边界提醒来完成。

5. **必须留下可迁移的骨架**
后续改写时，可复用的是“结构”和“语言动作”，不是作者个人经历或独家资源。

## 共享总骨架

`06_checklist_template` 的总骨架只有 5 段：

1. `Hook`
2. `Why it matters`
3. `How to get/use/build it`
4. `Proof / Validation`
5. `Boundary / Close`

不同内容的差异，主要发生在第 3 段和第 4 段。
所以后续不要把它拆成一堆新框架，而应该把第 3、4 段做成可选的结构参数包。

## 重要原则：结构是隐式骨架，不是外显模板

后续生成内容时，**绝对不能把下面的“稳定段落流”当成可见目录照抄输出**。

这些段落流的作用是：

- 约束信息组织顺序
- 确保关键功能段不缺失
- 帮助改写时判断什么该提前、什么该后置

它们**不是**：

- 文章必须显示出来的小标题列表
- 每篇都要按同样顺序明写的模板
- 一套可以机械复制的 AI 结构

### 正确用法

把 `稳定段落流` 当成后台控制骨架，而不是前台排版模板。

允许的操作：

- 合并相邻功能段
- 调整局部顺序
- 省略弱功能段
- 用场景、对话、案例、反问、清单、FAQ 等形式承载功能段
- 把一个功能段拆散到不同自然段中

不允许的操作：

- 每篇都按固定小标题顺序输出
- 每篇都用 “首先 / 其次 / 最后” 机械展开
- 每篇都把 `Why`、`How`、`Proof` 写成教科书式平铺段
- 只满足结构，不保留样本里的语感和节奏

### 最终要求

生成结果必须同时满足两件事：

1. **结构上有条理**
2. **表面上不像模板拼装**

也就是说，读者读起来要觉得“作者本来就是这么写的”，而不是看出背后有一套框架在硬撑。

## 结构参数轴

`06_checklist_template` 的变体不要继续无限增殖。统一只在下面 5 个参数轴上变化：

- `reader_state`
  - `cold_start`: 完全没开始
  - `using_but_blocked`: 已经在用，但遇到能力瓶颈
  - `system_builder`: 想搭一套系统或工作流
  - `option_overload`: 面对多个可选 skill / tool，不知道从哪下手

- `promise_type`
  - `onboard`
  - `extend`
  - `architect`
  - `curate`

- `proof_mode`
  - `step_by_step`
  - `before_after`
  - `components_and_roles`
  - `itemized_demo`

- `action_depth`
  - `single_action`
  - `guided_path`
  - `multi_component_system`

- `close_mode`
  - `validation`
  - `pitfalls`
  - `boundary_judgment`
  - `selection_advice`

## 改写时的双层控制

后续真正用于生成时，`06_checklist_template` 必须分成两层来控制：

### 第一层：隐式结构层

只负责回答：

- 这篇内容的核心承诺是什么
- 哪些功能段必须出现
- 证据应按什么逻辑进入
- 结尾应该如何收束

### 第二层：表层表达层

只负责回答：

- 这篇看起来像谁、像哪类样本
- 句子节奏应该快还是慢
- 是偏“保姆级”、偏“直接判断”、偏“系统化拆解”，还是偏“清单快扫”
- 该用场景、故事、提问、列表、FAQ、案例，还是对比来承载结构

**只有结构层，没有表层表达层，产物就会很 AI。**
**只有表层表达层，没有结构层，产物就会散。**

所以后续改写必须是：

`隐式结构层 + 样本语感层`

而不是：

`稳定段落流 = 最终文章结构`

## 语言风格学习原则

`06_checklist_template` 不是只学习“写什么”，还必须学习样本是“怎么写”的。

语言风格学习的对象不是具体句子，而是下面这些可迁移的风格特征：

- `tone`
  - 保姆级
  - 朋友式提醒
  - 工程师式拆解
  - 直接判断
  - 拟人化解释

- `sentence_rhythm`
  - 短句推进
  - 长短交替
  - 列表/清单密集
  - 场景对话切入

- `rhetorical_moves`
  - 先打痛点
  - 先交作业/先讲结果
  - 先讲“为什么原来不行”
  - 先给目录/总览降低焦虑
  - 用“装前/装后”建立对比
  - 用“我替你筛好了”代替解释成本

- `reader_distance`
  - 像教朋友
  - 像给同事交接
  - 像给圈内人讲系统设计
  - 像替读者做选择

### 风格迁移边界

可以学习：

- 语气
- 节奏
- 信息组织动作
- 场景切入方式
- 提醒与收尾方式

不能照抄：

- 原句
- 作者专属口头禅
- 个人经历细节
- 独家资源和身份背书
- 过强的个人人设表达

### 抗 AI 味要求

后续使用该父框架生成时，必须避免：

- 过于完整均匀的段落长度
- 每段都像“说明书”
- 机械性的总分总
- 反复出现标准转折词
- 清单、目录、小标题过度使用
- 每篇都长得像同一个模板换词

相反，应当主动允许：

- 某些功能段合并
- 某些功能段用一个案例带过去
- 先讲结果再补原理，或者先讲场景再补步骤
- 局部保留样本里的口语感、判断感、提醒感

## 4 组稳定结构参数包

下面 4 组参数包已经能覆盖当前 13 篇样本，不要再把它们当成平级大框架。

### A. guided_onboarding

#### 适用

- 安装、接入、配置、开通、完整攻略、入门路径
- 目标是把“不会”变成“能开始做”
- 读者最关心门槛、步骤和是否会踩坑

#### 对应样本

- VSCode + Copilot 保姆级教程
- OpenClaw 安装 + 接入 QQ
- OpenClaw 接入飞书
- 香港银行卡完整攻略
- Elasticsearch 学习路径
- OpenClaw 完全卸载指南

#### 稳定段落流

1. `Hook`
2. `Why this matters now`
3. `Prep / prerequisites`
4. `Step-by-step path`
5. `Validation / tips / FAQ`
6. `Boundary / close`

#### 常见钩子

- `保姆级`
- `一文讲全`
- `几分钟搞定`
- `0 基础也能做`
- `只看这篇就够了`

#### 证据组合

- 准备清单
- 操作步骤
- 命令 / 配置项
- 权限项 / 材料项
- 小技巧 / FAQ / 风险提醒

#### 语言动作

- 先降低门槛，再给步骤
- 先给大纲，再展开细节
- 多用动作导向句，而不是概念解释句
- 结尾补“适不适合你”或“有什么坑”

#### 改写禁区

- 只有步骤，没有告诉读者为什么值得做
- 只讲功能，不讲准备项
- 步骤太散，没有总览
- 全文没有验证/风险/边界提醒

#### 改写公式

`结果承诺 -> 进入门槛很低 -> 准备项 -> 步骤 -> 验证 -> 坑点/边界`

#### 表层表达偏好

- 更接近“保姆级带做”
- 常见外显形式：目录、大纲、步骤、FAQ、提醒
- 但不要求每篇都显式列目录或编号
- 如果 source 本身已经很具体，可直接用场景和动作句带步骤，不必强行列 1/2/3

### B. workflow_architecture

#### 适用

- 多 Agent 协作
- 系统设计
- 长时间运行机制
- 技术工作流或自治开发系统
- 某个系统的核心循环/架构拆解

#### 对应样本

- Agent Team
- OpenClaw 跨境电商团队
- Self-Improving Skills

#### 稳定段落流

1. `Hook / end state`
2. `Current pain or blind spot`
3. `Core principle`
4. `Components / roles / files`
5. `Orchestration flow`
6. `Proof / result`
7. `Pitfalls / safeguards`
8. `Close`

#### 常见钩子

- `我跑通了一套...`
- `最后一公里/最后一环`
- `5个AI员工`
- `连续运行 X 小时`

#### 证据组合

- 角色分工
- 文件 / 组件结构
- 数据结构 / 命令 / 参数
- 工作流示例
- 真实项目结果
- 失败点或 safeguard

#### 语言动作

- 用系统比喻替代抽象概念
- 先讲“为什么原来不行”，再讲新架构
- 组件和流程分开写
- 最后用结果或边界回收，不让文章停在抽象层

#### 改写禁区

- 只有架构图，没有真实运行结果
- 只有组件列表，没有串联流程
- 只有结果，没有解释为什么原方案不行
- 完全不提风险或 guardrails

#### 改写公式

`结果/盲点 -> 为什么旧方法失效 -> 新原则 -> 组件 -> 编排 -> 结果 -> 风险边界`

#### 表层表达偏好

- 更接近“系统设计拆解”
- 常见外显形式：问题定义、核心原则、组件说明、流程演示
- 允许穿插文件名、角色名、循环名等“结构锚点”
- 不要求把每个组件都写成标题，案例和组件可以交错出现

### C. capability_extension

#### 适用

- 给现有系统补一个新能力
- 装一个新 skill / 模块 / 接口后显著增强现有工具
- 重点不是从零开始，而是“装上之后立刻变强”

#### 对应样本

- 给 OpenClaw 装上搜索大脑
- Tushare skill 让 OpenClaw 做选股和热点追踪

#### 稳定段落流

1. `Current limitation`
2. `Why default state is not enough`
3. `What the extension changes`
4. `Install / integrate`
5. `Before / after payoff`
6. `Close`

#### 常见钩子

- `给 X 装上 Y`
- `让 X 变成...`
- `5分钟让你的 X 会做...`
- `装了之后到底能干什么`

#### 证据组合

- 场景对话
- 装前/装后差异
- 新能力列表
- 安装指令
- prompt 示例
- 自动结果描述

#### 语言动作

- 不先列功能，先写“不够用”的场景
- 用“前后差异”代替抽象性能描述
- 尽量把 skill 的价值翻译成业务动作
- 结尾把能力升维成“新器官/新大脑/新员工”

#### 改写禁区

- 只介绍 skill 名称，不告诉读者它补了什么洞
- 只有安装链接，没有装后场景
- 能力描述过散，没有前后差异主线

#### 改写公式

`现有不足 -> 新能力补洞 -> 怎么装 -> 装完能做什么 -> 为什么值`

#### 表层表达偏好

- 更接近“能力升级说明”
- 常见外显形式：场景痛点、装前/装后对比、功能包总结
- 安装部分宜短，不宜成为全文主体
- 重点是让读者感到“这个能力值得加”

### D. curated_list

#### 适用

- 多个 skill / tool / repo / prompt 的精选清单
- 文章价值来自筛选，而不是深入拆一个单点
- 读者目标是“先拿一组可以试的东西”

#### 对应样本

- 5个龙虾 Skill 开源分享

#### 稳定段落流

1. `Why this list matters`
2. `Item 1..N`
3. `Each item = what it solves + quick demo + action link`
4. `Selection advice / close`

#### 常见钩子

- `N 个`
- `最后一个最强`
- `我替你筛好了`
- `直接拿去用`

#### 证据组合

- 每项的 demo
- 安装指令
- 仓库链接
- 适用场景

#### 语言动作

- 高密度列举
- 每项都能立刻行动
- 用悬念词提高看完率
- 用“你不需要全懂，先试”降低心理成本

#### 改写禁区

- 清单里每项都没有行动入口
- 清单太长但没有筛选逻辑
- 所有条目都只写功能，没有场景

#### 改写公式

`为什么值得看 -> 按项快速展示 -> 每项都能立刻试 -> 收尾给选择建议`

#### 表层表达偏好

- 更接近“精选清单”
- 常见外显形式：条目化快扫、每项 demo + 安装入口 + repo
- 不追求深讲单项，而追求低决策成本
- 可以保留轻微悬念感，但不能为了悬念破坏信息效率

## 子包之间的边界

为了避免后续框架越堆越多，新增内容必须先问：

1. 这篇的核心目标是让读者**开始做**、**理解系统**、**增强现有能力**，还是**从多个选项中做选择**？
2. 它的证据主要是**步骤**、**架构组件**、**前后对比**，还是**条目清单**？
3. 它的中段核心是在讲**路径**、**系统**、**增强**，还是**筛选**？

只要这 3 个问题能落进上面 4 组参数包，就不要新建框架。

## 选择规则

如果 source 的主要承诺是“跟着做完”，优先选 `guided_onboarding`。

如果 source 的主要承诺是“理解并搭出一套系统”，优先选 `workflow_architecture`。

如果 source 的主要承诺是“给现有系统增加一个关键能力”，优先选 `capability_extension`。

如果 source 的主要承诺是“我替你筛好了一组可直接用的东西”，优先选 `curated_list`。

## 改写总检查表

在任何一个参数包下改写，都必须检查：

- 有没有明确读者收益
- 有没有可执行动作
- 有没有具体证据支撑
- 有没有降低试错成本
- 有没有边界提醒或适用范围
- 有没有保留目标样本的语气和节奏，而不是只保留结构
- 有没有避免把隐式骨架直接翻译成外显模板

如果改写后的文章只剩“功能介绍”或“抽象评价”，说明已经偏离了 `06_checklist_template`。

## 当前结论

`06_checklist_template` 不应该继续被理解成“技术分享”这个宽泛标签。

更准确的理解是：

**一个父框架：能力交付型技术分享**

**4 组稳定结构参数包：**

- `guided_onboarding`
- `workflow_architecture`
- `capability_extension`
- `curated_list`

后续如果要做机器可读的 `FRAMEWORK_SPEC.json`，应当直接把这 4 组参数包内嵌进去，而不是继续平铺成更多平级框架。

## 配套文件

- 结构与改写规则：`FRAMEWORK.md`
- 单篇样本拆解：`SAMPLE_DECOMPOSITIONS.md`
- 风格画像：`STYLE_PROFILES.md`

后续使用顺序应当是：

1. 先用本文件确定父框架和结构参数包
2. 再看 `STYLE_PROFILES.md` 决定“怎么说”
3. 必要时回看 `SAMPLE_DECOMPOSITIONS.md` 校验样本依据
