# T01 单泳道试点契约 v1

## 1. 目标与边界
- 泳道ID: `T01_release_decode`
- 目标: 把发布类素材转成“为什么值得看 + 谁该行动 + 怎么行动”的决策型文章。
- 非目标: 新闻复述、功能罗列、品牌宣传稿、无证据观点文。
- 试点范围: 只做 T01，一次只验证这一条泳道。

## 2. 输入契约
- writer 输入固定为三件套: `topic_card + source_bundle + framework_spec(T01)`
- 不允许只用 `topic_card` 直接写文。
- 不允许 source 缺证据时强行产文。

## 3. 写作比例与模式
- `composition_mode = mixed`
- `generation_ratio = 0.8`
- `rewrite_ratio = 0.2`
- 20% 改写必须承载: 发布事实、关键参数、限制条件、官方表述边界。
- 80% 生成用于: 价值判断、影响拆解、行动建议、适用边界。

## 4. 最低 source 要求（硬门槛）
- 至少 2 条 `primary_sources`:
- 其中至少 1 条来自官方发布位（官方 X / 官方文档 / release note / changelog）。
- 另一条必须能补“变化细节”或“限制条件”，不能只是转述。
- 至少 1 条 `supporting_source`（开发者实测、社区反馈、配套仓库、FAQ 等）。
- 至少 5 个可核验 `fact_anchors`（参数、时间、版本、入口、限制、定价等）。
- 每个 `fact_anchor` 必须能回指到具体 source URL。

## 5. 失败即拦截（Fail-fast）
- 只有“发布了”但没有“变了什么”。
- 只有单一来源，且无补充证据。
- 找不到可验证链接或发布时间。
- 无法明确“谁会受影响”。
- 事实锚点少于 5 条。

## 6. 文章结构要求
- 标题: 必须包含“变化判断”，不能是公告标题改写。
- Dek: 必须回答“真正值得看的变化是什么”。
- 开篇前两段: 必须出现发布动作 + 高价值判断。
- 中段: 必须出现“变化前后对比 + 人群影响 + 不该用边界”。
- 结尾: 必须输出可执行动作（立刻试用 / 观察 / 暂缓）。

## 7. 质量检查（5项硬检查）
- 检查1: 是否讲清“具体变化”而非口号。
- 检查2: 是否至少引用 5 个 fact anchors。
- 检查3: 是否讲清“谁该行动、谁不该行动”。
- 检查4: 是否包含“限制/代价/风险”段。
- 检查5: 是否避免公告复读（相似度阈值可后配）。

## 8. 试点产能建议
- 每批先处理 20-40 条源。
- 聚类后保留 8-12 个 topic。
- T01 只取 Top 3-5 进入写作。
- 每次先产 3 篇，人工复盘后再扩量。

## 9. 与旧架构关系
- 旧 prefilter/router 可暂时弱化。
- 本试点以“topic 优先 + T01 契约强约束”为主。
- 等 T01 稳定后，再决定是否把同样模式推广到其他泳道。
