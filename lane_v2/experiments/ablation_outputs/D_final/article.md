# AI agent 跑了一整晚挂掉，你第二天才知道——这个插件补上了这个缺口

> Claude-to-IM-Skills 新增微信支持，扫码配置，让 Claude Code / Codex 的长任务进度直接推送到你手机上

> 你的 AI agent 跑了一整晚，挂了——你第二天早上才知道。

这是用 Claude Code 或 Codex 跑长任务的人都踩过的坑。把任务丢给 agent，锁屏睡觉，结果半夜报错退出，没有任何通知，第二天打开电脑看到一个失败的任务和空白的进度。**这个痛点的本质，是 agent 在跑、但你跟它断连了。**

归藏（@op7418）在 3 月 24 日发布了一条更新：之前用于把 Claude Code 和 Codex 连接到飞书 IM 的插件 Claude-to-IM-Skills，现在支持接入微信了。配置方式很直接——插件会自动打开一个网页，你用微信扫码即可完成授权，全程不需要手动填写 token 或修改配置文件。

## 这个插件解决的是什么问题

用 Claude Code 跑长任务的开发者，过去只有两个选项：坐在屏幕前盯着，或者全程脱机等结果。飞书版本的 Claude-to-IM-Skills 已经让一部分人有了任务推送通知，但飞书在中国开发者中的普及率远不及微信——很多人的手机主力通知入口是微信，而不是另一个需要单独开着的 IM。

微信支持的加入，意味着这条链路终于打通了：**Claude Code 或 Codex 在本地或云端执行任务 → 任务状态变更时通过插件推送消息 → 你的微信收到通知**。不需要开着电脑，不需要盯着终端，你可以去做别的事，任务有进展了手机自然会响。

## 怎么接入，门槛有多低

根据归藏分享的信息，接入流程的核心操作就一步：**插件启动后会自动打开一个授权网页，你扫码，完成**。这相比之前手动配置 webhook 地址、填写 bot token 的方式，摩擦低了很多。

适合现在就去接的场景：

- 用 Claude Code 做代码库重构，任务会跑几十分钟到数小时，想知道什么时候完成或报错
- 用 Codex 完成端到端功能开发（比如 Notion 工程师 Ryan Nystrom 用 Codex 独立完成了 AI Voice Input 功能的 web 和 desktop 迁移），任务周期长，需要在移动端跟进状态
- 在无人值守的服务器上跑批量任务，希望任务结束或异常时第一时间收到消息

## 边界和你应该注意的地方

几个需要清醒的地方：

**微信个人号接入有平台风险。** 微信并没有向第三方开放稳定的消息推送 API，通过扫码授权的接入方式，依赖的是非官方通道。这意味着随时可能因微信客户端更新或策略调整而中断，不适合用于生产环境的关键告警链路。**如果你的任务失败会造成实际损失，建议同时保留一个稳定的 fallback 通知方式**（比如飞书版本，或者 email 告警）。

**这个插件解决的是"通知断连"问题，不是任务恢复问题。** 它告诉你任务挂了，但任务本身还是挂了。想让长任务有断点续跑能力，还需要另外的机制（比如 Claude Code 的 auto 模式——它引入了一个分类器来判断操作风险，有破坏性风险的操作会被拦截并换一种方式完成，官方坦承仍有边界）。两个工具解决的是不同层面的问题。

**插件早期更新频率可能较高。** 从飞书版本到现在加入微信支持，说明这个项目仍在快速迭代，接入前最好确认版本文档是最新的。

## 现在可以做什么

如果你在用 Claude Code 或 Codex 跑耗时超过 30 分钟的任务，且没有实时通知手段，这个时间点接入微信版本的 Claude-to-IM-Skills 是一个低成本的改善动作。

**具体步骤**：找到归藏（@op7418）发布的原帖，进入 Claude-to-IM-Skills 的 GitHub 仓库，按最新文档完成微信扫码授权配置，然后跑一个短任务验证推送是否正常。整个接入过程按现有描述应该在十分钟以内。

考虑到微信接入的平台稳定性问题，建议把它当成**主力便捷通知 + 飞书/邮件保底备份**的组合来用，而不是单点依赖。
