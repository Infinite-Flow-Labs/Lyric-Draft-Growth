# 06_checklist_template 风格画像

说明：本文件不是结构模板，而是基于 13 篇真实样本阅读后归纳出的“怎么写”画像。

用途：

- 给后续改写提供语感约束
- 避免只学结构、不学表达
- 避免把 `稳定段落流` 直接翻译成模板化文章

---

## 总体观察

`06_checklist_template` 的共同风格，不是“像教程文”这么简单，而是：

- **都在替读者压缩决策成本**
- **都很少从抽象定义开始**
- **都倾向于先给结果感、场景感或问题感**
- **都把复杂能力翻成可执行动作**
- **都保留较强的人说话感，而不是说明书口吻**

从样本上看，真正可迁移的风格不是某一句话，而是这些动作：

- 先打读者当下痛点
- 先交作业，再讲原理
- 先让读者觉得“我也能做”
- 先替读者完成筛选，再让他行动
- 先把系统讲成人能理解的角色、流程或场景

## 1. guided_onboarding

### 样本依据

- `2029894874888278198` VSCode + Copilot 保姆级教程
- `2031396119104856437` OpenClaw 安装 + 接入 QQ
- `2032432901812588603` OpenClaw 接入飞书
- `2030878769955901844` 香港银行卡完整攻略
- `2029203445589414147` Elasticsearch 学习路径
- `2031555224968785990` OpenClaw 完全卸载指南

### 语气核心

- 像一个熟门熟路的人在带新人上手
- 不是冷静中立，而是带一点热情、提醒和替读者省事的姿态
- 中文样本里普遍有“朋友式教学感”和“代劳感”

### 开头动作

常见不是直接定义主题，而是这几种开头：

- 先承诺：`保姆级 / 一文讲全 / 几分钟搞定`
- 先交代背景热度或现实问题
- 先说“你不用担心门槛”
- 先给目录、大纲或步骤总览，降低焦虑
- 少数样本会用小故事或角色对话把新概念拉近

### 句子节奏

- 整体偏快
- 短句密度高
- 感叹和反问明显比其他子模式多
- 经常“短句推进 + 小列表插入”
- 读起来像人边讲边提醒，不像论文

### 常见表层承载形式

- 大纲
- 目录
- 步骤
- FAQ
- 小技巧
- 注意事项
- “如果你只想解决问题，直接看这段”

### 语言动作

- 先把门槛压低：`完全不会也行`、`无脑跟着做就行`
- 会主动替读者预判顾虑：贵不贵、麻不麻烦、会不会失败
- 喜欢把复杂词翻成生活化说法
- 收尾经常补一句边界判断：适不适合你、会不会吃灰、值不值得长期用

### 最容易出 AI 味的地方

- 把所有步骤平均铺开，变成说明书
- 每段都很工整，没有“人味提醒”
- 只写步骤，不写为什么值得做
- 结尾没有边界提醒，只剩“总结一下”

### 改写时应保留

- 朋友式教学感
- 降门槛动作
- 步骤前的总览感
- 中途提醒和小技巧
- 最后的边界判断

### 改写时应避免

- 所有文章都变成 `准备 -> 步骤1/2/3 -> 总结`
- 每篇都用同一套编号标题
- 把样本里的口语感洗没

### 一句话画像

像一个已经踩过坑的人，在帮你把第一次上手的心理负担和试错成本一起拿掉。

## 2. workflow_architecture

### 样本依据

- `2030113154739621909` Agent Team
- `2030123309594259506` 多 Agent 跨境电商团队
- `2033261636195373123` Self-Improving Skills
- `2031412763902534069` Cloudflare PR agents

### 语气核心

- 更像工程师在做系统复盘或架构拆解
- 情绪浓度低于 onboarding，但判断性更强
- 会先讲“原来为什么不行”，再讲“我现在怎么做”
- 整体更像在交一份系统设计作业

### 开头动作

常见是三类：

- `先说结论 / 我跑通了一套`
- `最后一环 / 最后一公里 / 盲点`
- `一个团队一周的活，现在一杯咖啡时间就跑完`

也就是说，开头通常不是“今天分享一个工作流”，而是先亮出结果或盲点。

### 句子节奏

- 平均句长明显更长
- 解释句占比高
- 常见“短句打痛点 + 长句解释架构”
- 会出现结构锚点：文件名、角色名、step 名、系统名
- 整体节奏是“先压问题，再展开系统”

### 常见表层承载形式

- 组件说明
- 角色拆分
- 工作流 step
- 系统原则
- 配置片段 / 数据结构示例
- 真实运行结果
- safeguard / pitfalls

### 语言动作

- 先定义问题，不直接堆方案
- 把架构讲成人能理解的模型：轮班工人、团队协作、两阶段流程
- 组件和流程会分开写
- 结果段往往很重要，用来证明不是纸上谈兵

### 最容易出 AI 味的地方

- 把架构写成百科条目
- 只有组件列表，没有运行中的动态感
- 只有“这个系统包含 A/B/C”，没有“为什么这样设计”
- 没有结果验证，只剩空洞原则

### 改写时应保留

- 明确的问题定义
- 核心设计原则
- 架构锚点
- 运行中的动态感
- 最后的 guardrails 或边界判断

### 改写时应避免

- 用空泛的系统语言替代具体例子
- 每篇都写成“背景-方案-优势-结论”的标准 PPT 腔
- 强行把所有组件都列成小标题

### 一句话画像

像一个真的把系统跑起来的人，在向同类解释它为什么能跑、怎么跑，以及哪里最容易翻车。

## 3. capability_extension

### 样本依据

- `2031965027314647316` 给 OpenClaw 装上搜索大脑
- `2031552504593526946` Tushare skill

### 语气核心

- 不是从零教学，而是“现有状态不够用，我给你补一块”
- 比 onboarding 更像升级说明，比 architecture 更贴着使用场景
- 经常有一种“你装上这个，就突然会做某件事了”的语气

### 开头动作

常见开头是：

- `你的 X 是否也有这样的困境`
- `装了之后到底能干什么`
- 先给读者一组失败场景
- 先把旧状态写得不够爽，再提出扩展能力

### 句子节奏

- 句子最短
- 场景切换很频繁
- 经常用对话、提问、前后对比来推进
- 不是长解释，而是“痛点一刀刀打进去”

### 常见表层承载形式

- 场景一 / 场景二 / 场景三
- 装前 / 装后
- 能力列表
- prompt 示例
- 安装入口
- 收益归纳

### 语言动作

- 不先讲功能定义，先讲“不装时有多难受”
- 用“新大脑 / 新超能力 / 新员工”这类能力隐喻升维
- 安装动作通常很短，主体重点放在“值不值得装”
- 经常把技术能力翻译成业务动作或日常结果

### 最容易出 AI 味的地方

- 只剩“这个 skill 支持什么功能”
- 痛点写得像功能缺陷列表，没有真实使用感
- 安装动作过长，冲淡了能力增强主线
- 结尾只是“欢迎使用”

### 改写时应保留

- 明确的旧状态不足
- 装前/装后差异
- 技术能力翻译成使用场景
- 最后的能力升维句

### 改写时应避免

- 把全文写成 README 摘要
- 一股脑罗列功能点
- 场景和安装顺序颠倒，让读者还没觉得值就先去装

### 一句话画像

像在告诉一个已经上手工具的人：“你卡住的不是不会用，而是还没给它装上这块关键器官。”

## 4. curated_list

### 样本依据

- `2033803842668949834` 5 个龙虾 Skill 开源分享

### 语气核心

- 不是“我来全面教你”，而是“我替你筛好了”
- 作者位置更像 curator，不像老师
- 强调选择效率，而不是单项深度

### 开头动作

- 先讲为什么现在值得看这个清单
- 先给一点个人判断或圈内判断
- 很快进入列表，不长篇铺垫

### 句子节奏

- 最碎、最快
- 条目切换快
- 段落很短
- 高频使用“演示一句话 + 安装入口 + 链接”组合

### 常见表层承载形式

- 列表
- demo 句
- repo 链接
- 安装指令
- 最后一个更强/更有记忆点

### 语言动作

- 不解释太多原理
- 更强调“你可以直接试”
- 一项一项给行动入口
- 轻微保留悬念和推荐感

### 最容易出 AI 味的地方

- 清单里每一项长度完全一样
- 每项都按同样句式机械展开
- 只有“功能介绍”，没有“我为什么选它”
- 悬念写得太假

### 改写时应保留

- 低决策成本
- 高行动密度
- 每项都有入口
- curator 的筛选姿态

### 改写时应避免

- 把清单写成综述长文
- 每项都解释过深
- 过度整齐，像数据表格转 prose

### 一句话画像

像一个已经试过一圈的人，把真正值得先上手的几样东西捞出来递给你。

## 子模式之间最关键的风格差异

如果只看结构，4 个子模式容易被误判成“都是技术分享”。  
真正区分它们的，是下面 4 个维度：

- `reader_distance`
  - `guided_onboarding`: 像带新手
  - `workflow_architecture`: 像给同类工程师做系统拆解
  - `capability_extension`: 像给现有用户补能力
  - `curated_list`: 像替读者筛选

- `opening_pressure`
  - `guided_onboarding`: 先减压
  - `workflow_architecture`: 先抬结果或暴露盲点
  - `capability_extension`: 先放大现有不足
  - `curated_list`: 先建立“值得看这份清单”

- `surface_form`
  - `guided_onboarding`: 大纲、步骤、FAQ、提醒
  - `workflow_architecture`: 原则、组件、流程、结果
  - `capability_extension`: 场景、对比、安装、收益
  - `curated_list`: 条目、demo、repo、安装入口

- `close_style`
  - `guided_onboarding`: 边界提醒 / 使用建议
  - `workflow_architecture`: 结果验证 / safeguard
  - `capability_extension`: 能力升维 / 效率收益
  - `curated_list`: 行动 CTA / 选择建议

---

## 最终结论

后续改写 `06_checklist_template` 时，不能只问：

- 用哪个结构参数包？

还必须问：

- 该站在什么位置对读者说话？
- 该用什么表层形式承载结构？
- 该保留什么样的句子节奏？
- 该避免哪一种模板味？

只有这样，生成结果才会：

- 有结构
- 有条理
- 但不像套模板
