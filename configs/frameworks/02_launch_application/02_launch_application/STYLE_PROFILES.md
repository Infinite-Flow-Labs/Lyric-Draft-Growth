# 02_launch_application 风格画像

说明：本文件基于 8 篇真实样本重读后归纳“怎么写”，而不是“写哪些内容”。

用途：

- 给后续改写提供采用逻辑之外的表层表达约束
- 避免把发布应用文写成统一教程模板
- 避免只学“很新很强”，不学“怎么让人真的愿意用”

---

## 总体观察

`02_launch_application` 的共同风格，不是“帮别人做发布会复述”，而是：

- 先把发布信号抓住
- 立刻翻译成读者能理解的价值变化
- 再决定是应该给 quickstart、给场景、还是给纠偏

从样本上看，真正可迁移的风格动作是：

- 不在发布消息上停留太久
- 很快从“发生了什么”切到“你能怎么用”
- 有意识地降低理解门槛，常用白话定义
- 能举步骤就举步骤，能举场景就举场景，能举误区就举误区
- 结尾一般会把读者推向某个采用动作，而不是只留下感叹

## 1. feature_playbook

### 样本依据

- How to run subagents in Codex
- Claude 给了你 1M Token 的上下文，但别真的全塞满
- You Should Be Using Subagents in Codex!

### 语气核心

- 像一个真正用过新 feature 的人，在给你做高密度说明
- 有主张，但不是空喊“很强”
- 会主动帮读者避坑

### 开头动作

常见动作：

- 先抛发布事实，然后立刻接“你真正需要知道的是...”
- 先承认这次更新是好事，再立刻补一个限制条件
- 直接定义功能本质，不兜圈子

### 句子节奏

- 三个子模式里句子最长、逻辑密度最高
- 英文样本更像精炼文档转写
- 中文纠偏样本会用反问和短句打断节奏，避免变成讲义

### 常见表层承载形式

- 概念定义
- Example prompt / config 片段
- Best use cases / tips
- `when not to use`
- 最后附一个 quick action

### 语言动作

- `what it is`
- `how to use`
- `best use cases`
- `when not to use`
- `别急`
- `道理其实很简单`

这类文章很依赖“解释动作”，不是很依赖情绪渲染。

### 内部常见口吻分支

- `instructional`
  - 更像高密度上手说明
  - 代表样本：1、7

- `corrective`
  - 更像借发布消息做一次采用纠偏
  - 代表样本：5

这两个分支不需要拆成新框架，只需要在改写时调 `stance`。

### 最容易出 AI 味的地方

- 写成“第一，第二，第三”的死板教程
- 只有 feature list，没有使用判断
- 全文太像官方 help center
- 纠偏文写成假装理性的鸡汤总结

### 改写时应保留

- 直接性
- 白话解释
- 采用边界
- 让读者尽快知道第一步该做什么

### 改写时应避免

- 拖很久才讲“到底怎么用”
- 用大量抽象术语堆出专业感
- 只讲概念，不讲动作

### 一句话画像

像一个已经把新 feature 玩明白的人，替你把“新消息”压缩成“可执行用法”。

## 2. pain_to_adoption

### 样本依据

- 90%的亚马逊选品还不如算命，AI选品系统发布！！
- 龙虾专属模型来了，给你的虾换个好脑子吧
- 这应该使用 OpenClaw 最简单的方案了，点击安装就能用

### 语气核心

- 像一个已经被旧方案折腾够了的人，在告诉你“终于有个顺手答案了”
- 比 `feature_playbook` 更口语、更高对比、更带采用情绪
- 不是纯卖货，而是“我知道你卡在哪，所以我先替你把旧痛点说透”

### 开头动作

- 直接从“旧方法太痛苦”开场
- 先说一个行业里人人都懂但没人讲明白的破事
- 先给结论：这次终于简单了 / 终于像样了 / 终于能干活了

### 句子节奏

- 平均段落最短，但段落数最多
- 中文短段很多，推进快
- 冒号、编号、问答、比喻都比较多
- 情绪明显高于另外两类

### 常见表层承载形式

- 痛点场景
- old vs new 对照
- 安装步骤 / walkthrough
- 工作流拆解
- 真实任务测试
- 套餐、价格、注册链接

### 语言动作

- `说实话...太费劲了`
- `真正的问题是...`
- `这才是...`
- `终于...`
- `马上给大家安排`
- `你的 X，值得一个好 Y`

这类文章的关键不是“会不会讲步骤”，而是“旧痛点有没有讲透，采用冲动有没有建立起来”。

### 最容易出 AI 味的地方

- 只有夸张形容词，没有真实采用路径
- 看起来像硬广，不像踩坑转译
- old/new 对照过于模板化
- 结尾只有 CTA，没有证明

### 改写时应保留

- 旧基线的失败感
- 新方案的顺滑感
- 真实任务或步骤证明
- 采用后的具体想象

### 改写时应避免

- 一上来就吹产品能力
- 用“效率提升明显”这种空话替代真实改善点
- 只剩下载链接，没有体验过程

### 一句话画像

像一个被旧方法折磨过的人，终于拿到了能真正把事做成的新方案，于是带着你一路装到能跑起来。

## 3. release_showcase

### 样本依据

- 清华团队开源 AI 课堂，人人都能学起来
- 《OpenClaw 从入门到精通指南》正式发布，开源免费！

### 语气核心

- 更像热情、可信的资源分享
- 重点不是完整教程，而是先让你快速理解“这东西值不值得进收藏夹”
- 热情里带具体，不走空泛夸赞

### 开头动作

- `开源了个好东西`
- `今天，终于正式和大家见面了`
- `简单说，它是...`
- `我们打磨了很久`

### 句子节奏

- 三类里最短、最轻
- 常用短段切场景
- 情绪是热的，但推进不凶
- 几乎不用复杂理论解释

### 常见表层承载形式

- 一句话定位
- 内容清单 / 能力概览
- 场景举例
- 制作过程或亲测背书
- 最后给入口链接

### 语言动作

- `简单说`
- `先说它怎么用`
- `更有意思的是`
- `比如`
- `我希望分享给你的是...`

这一类很依赖“把抽象东西变成可以想象的生活或工作场景”。

### 最容易出 AI 味的地方

- 写成空泛通稿
- 场景全是编出来的万能例子
- 只有“很强很全很免费”，没有对象定位
- 制作过程写成煽情自述，丢掉信息价值

### 改写时应保留

- 低门槛定义
- 具体场景
- 真实制作或体验背书
- 最终清晰入口

### 改写时应避免

- 用长篇抽象介绍拖慢节奏
- 把资源分享写成产品白皮书
- 只顾热情，不给读者判断抓手

### 一句话画像

像一个真的把资源试过、整理过的人，在告诉你：这个东西值得点开，而且我先替你想好了它能怎么用。

---

## 子模式之间最关键的风格差异

- `feature_playbook`
  - 更像解释和纠偏
  - 重心在“怎么正确使用”
  - 逻辑密度最高

- `pain_to_adoption`
  - 更像踩坑后的采用转译
  - 重心在“为什么旧方案不行、这次为什么顺”
  - 口语感和推进感最强

- `release_showcase`
  - 更像资源或能力展示
  - 重心在“为什么值得试”
  - 最轻、最顺、场景化最多

## 最后约束

后续生成 `02_launch_application` 时，不能只学“发布后应该写 6 段”，必须同时学：

- 什么时候该解释
- 什么时候该先打旧痛点
- 什么时候该用多个场景让能力变得可感
- 什么时候该把热情压住，补上边界

否则生成结果会要么像官方教程，要么像通稿，要么像 AI 拼接出来的产品介绍。
