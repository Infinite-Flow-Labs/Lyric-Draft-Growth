# T01 发布解读型（Release Decode）

## 1. 类型定义
- 类型ID：`T01_release_decode`
- 目标：把“发布消息”转成“用户为什么现在要关注、如何判断是否接入”的决策型文章。
- 非目标：新闻播报、功能罗列、产品宣传稿、纯转述。

## 2. 适用来源
### 2.1 必需来源（至少满足一项）
- `official_x` 的发布相关帖子（模型/产品/能力/API/政策更新）。
- 官方 changelog / release note 页面（若抓到 linked context）。
- 官方文档更新中包含“新能力、新限制、新接口”的条目。

### 2.2 推荐补充来源（可选）
- 同一发布对应的技术说明页、参数表、定价页、FAQ。
- 同一发布的开发者账号补充说明（线程/答疑）。

### 2.3 禁止作为主来源
- 无明确发布动作的观点帖。
- 纯转发、纯情绪表达、无事实锚点短帖。
- 二手搬运且无法追溯原始链接的内容。

## 3. 输入契约（source_item 最低要求）
必须可从 `source_item.json` 提取以下字段：
- `canonical_url`
- `published_at`
- `title`
- `content.full_text`
- `content.primary_text_source`
- `extracted_signals.release_signals`
- `extracted_signals.fact_anchors`
- `source_assets`（用于确认是否有 linked context）

最低门槛：
- `release_signals` >= 1
- `fact_anchors` >= 3（有 linked context）或 >= 5（无 linked context）
- `full_text_words` 达到 gate 下限（公告类可放宽，但不能低于可解释阈值）

## 4. 漏斗规则（类型级）
## 4.1 Discovery 规则
- 保留 `post_kind=original` 为主。
- reply/retweet 仅在包含新增事实锚点时纳入。
- 对同一发布主题做去重，避免同源多条重复占位。

## 4.2 Type Gate（准入）
### 硬性通过条件（全部满足）
- 有明确“发布动作”证据：如发布词、版本词、上线词之一。
- 有至少一条“变化内容”可被复述（不是口号）。
- 有至少一条“用户影响面”可被复述（能力、限制、成本、流程之一）。

### 硬性拦截条件（任一命中）
- 全文只有“发布了”但无变化细节。
- 事实锚点不足，无法支撑非泛化改写。
- 无法定位原始链接或发布时间不明。

### 软性评分项（用于排序，不用于一票否决）
- 是否包含明确数值（速度、价格、上下文长度、支持范围等）。
- 是否包含“可执行入口”信息（API、产品入口、文档入口）。
- 是否包含“旧方案 -> 新方案”的对照线索。

## 4.3 路由策略（framework/submode）
主路由：
- `02_launch_application / release_showcase`

候选路由：
- `02_launch_application / feature_playbook`（当 source 已给出具体操作路径）
- `03_opinion_decode / signal_decode`（当 source 主要是“这意味着什么”，而非功能细节）
- `01_money_proof / metric_postmortem`（仅当发布同时给出强指标与结果）

路由约束：
- 不允许因为“有发布词”就自动落 `release_showcase`。
- 如果“应用价值信号”弱，必须由 reviewer 质疑并降级。

## 5. 写作生产模式
- 生产模式：`生成解读为主（80%） + 证据改写（20%）`
- 必须保留：
  - 发布事实锚点（人名/产品名/版本/时间/关键指标）
  - 变化点（到底变了什么）
  - 影响判断（谁会受影响、为什么）
- 生成部分必须承担：
  - 价值判断（为什么这次值得看）
  - 人群影响拆解（谁该用、谁先别用）
  - 行动建议（现在可以怎么做）
- 禁止：
  - 宣传话术改写成“看似中立”的宣传稿
  - 没有证据的“将改变行业”式判断

## 6. 文章结构要求（T01专用）
- 标题：必须体现“旧判断失效 / 新判断成立”中的至少一项。
- Dek：必须回答“这次发布真正值得看的是什么”，不能复述标题。
- 开篇（前2段）必须给出：
  - 具体发布动作
  - 一个高价值判断（不是背景介绍）
- 中段必须覆盖：
  - 变化内容拆解
  - 对用户流程或决策的影响
  - 至少一个“何时不该用/暂不接入”的边界
- 结尾必须输出：
  - 可执行判断（收藏观察 / 立即试用 / 暂缓接入）

## 7. 验收标准（Review Checklist）
- 是否能在30秒内说清“发布了什么变化”？
- 是否能在30秒内说清“这变化为什么重要”？
- 是否明确指出“谁应该行动、谁不必行动”？
- 是否有至少3个可核验事实锚点？
- 是否避免“新闻摘要味”和“产品宣传味”？

## 8. 失败模式（优先排查）
- 路由正确但文章仍像公告复述：说明 `opening_value_packet` 和 `mid_reset_plan` 未生效。
- source 很短却被放行：说明 gate 对公告类阈值过宽。
- 标题有钩子但正文平：说明中段重抓策略没有落地。
- 结尾空泛：说明 `closing_carry_packet` 约束不足。

## 9. KPI（类型级）
- `T01_throughput`: 每日通过 gate 的 T01 候选数。
- `T01_route_precision`: 路由到 `release_showcase` 后被人工判定“匹配”的比例。
- `T01_readability_pass`: 人工抽检“不是公告复述”的通过率。
- `T01_actionability_pass`: 人工抽检“结尾可执行判断”通过率。
