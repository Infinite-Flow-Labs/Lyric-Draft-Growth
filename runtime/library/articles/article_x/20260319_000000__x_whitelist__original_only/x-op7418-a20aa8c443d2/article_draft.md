# 别再把 Skills 当 Markdown 了：真正有用的是把团队经验做成会执行的文件夹

*Anthropic 内部已经把 Skills 用到了数百个场景，API 文档、测试验证、部署流程都能装进去。关键不在“写一份说明”，而在把脚本、数据、钩子和保护机制一起打包，让 Claude 真正能做事。*

很多人第一次听到 Skills，脑子里冒出来的还是一份 Markdown 文档。但这次真正值得记住的，不是“多了一种写法”，而是 Anthropic 已经把它用成了另一种东西：一个能被代理发现、读取、执行，还能在特定时机触发钩子的文件夹。内部现在有数百个 Skills，从 API 文档到部署流程都在覆盖，说明它早就不是笔记本级别的附录，而是团队经验的可执行封装。

这也是它最有意思的地方。Skills 里可以放脚本、资源文件、数据，甚至注册钩子函数。也就是说，它不是把知识写给人看，而是把知识做成 Claude 可以直接用的结构。很多时候，价值不在 README 那几行概述，而在文件夹里的渐进式披露：先读概览，需要时再深入到 examples、scripts、reference、gotchas 这些更细的层次。

Anthropic 对内部 Skills 的整理也很直接：好的 Skills 往往能清楚落进九类。第一类是库与 API 参考，解决的是 Claude 老是搞错某个库、CLI 或 SDK 的用法；第二类是产品验证，专门描述怎么测试或验证代码是否真的跑通，常常和 Playwright、tmux 这类工具绑在一起；第三类是数据获取与分析，把数据源、仪表盘和常用查询一起接进来；后面还有业务自动化、代码脚手架、代码质量与审查、CI/CD 与部署、运行手册、基础设施运维。分类本身不是为了好看，而是为了让每个 Skill 都指向一个明确的问题。

这个判断很关键。一个混乱的 Skill 往往什么都想装，最后什么都不够深；一个好的 Skill 则会非常明确地回答一件事。比如 billing-lib 这种内部计费库参考，不只是讲接口，还要把边界情况和常见坑写清楚；internal-platform-cli 这种内部 CLI 指南，得连每个子命令适合什么场景都交代明白。再比如 frontend-design，它不是让 Claude “了解设计”，而是让它更懂你们自己的设计系统。

最容易被低估的，是产品验证这类 Skills。很多团队会花大力气写功能，却把验证留给临时性的人工操作。Anthropic 的判断正好相反：验证 Skills 极其重要，甚至值得工程师花一周时间专门打磨。你可以让 Claude 录制测试视频，也可以在每一步强制执行程序化断言；无头浏览器、TTY 交互、Stripe 测试卡这些工具，都可以变成 Skill 里的脚本。signup-flow-driver、checkout-verifier、tmux-cli-driver 这类名字看起来很朴素，但它们真正做的是把“这东西应该能跑”变成可复现的动作。

真正把 Skills 从文档拉向系统能力的，是那些最佳实践。第一条就很实用：把 gotchas 写明白。Claude 会认真读这些内容，很多重复犯错其实不是不会做，而是陷阱没被明确写下来。第二条是利用文件系统做渐进式披露，不要把所有信息硬塞进一个 Markdown 文件。第三条是把可复用的脚本和辅助库放进去，让 Skill 自己带着工具箱，而不是每次都重写一遍。第四条更像工程化的底层能力：Skills 可以借助 `${CLAUDE_PLUGIN_DATA}` 做稳定存储，把上次运行状态、用户偏好、缓存数据、历史记录留住，慢慢形成“记忆”。

再往前一步，就进入了保护机制。对于删除数据、部署到生产环境这类危险操作，Anthropic 提倡用按需钩子，在执行前弹出确认提示，让用户明确批准。frontmatter 里就能配置 on-demand hooks。与此同时，PreToolUse 钩子还能用来度量 Skill 的使用情况，知道哪些最常用、哪些需要改进、哪些可以下线。也就是说，Skills 不是静态文档，它本身就能被治理。

怎么分发，也是 Skills 和普通文档不一样的地方。小团队可以把它们签进代码仓库，放到 `./.claude/skills` 里，跟代码一起提交；大团队则更适合内部插件市场，让用户按需安装，不污染所有人的上下文。Anthropic 的做法不是一开始就搞统一中心化管理，而是先让有用的 Skills 在沙箱区域里自然浮现，再慢慢迁到正式推荐区。这个思路很务实：先让价值出现，再谈标准化。

他们给出的管理建议也很像真正做过的人会说的话。要给每个 Skill 配清晰的 README 和使用示例，否则再好的功能也很难被采用。要定期清理没人用的 Skills，不然市场会膨胀到让人不知道选哪个。还要给反馈入口，GitHub Issues 也好，Slack 频道也好，总之让问题能被持续收回来。这些听起来不像“AI 叙事”里最光鲜的部分，但恰恰是决定它能不能落地的部分。

内部案例也说明了它到底值不值得做。commit-helper 负责把 Git 提交信息写得符合规范，几乎是每天都会用到的高频工具；pr-reviewer 自动化代码审查，先把风格、覆盖率、安全问题这些机械检查做掉，让人把注意力留给逻辑和架构；incident-response 则把生产事故响应流程变成清晰步骤，在紧急情况下尤其有价值；api-docs 让 Claude 调内部 API 时不必每次都翻文档或问人；test-runner 则把单元测试、集成测试、端到端测试的标准流程收拢起来。

这些例子背后，其实在回答同一个问题：什么样的重复劳动，值得被硬化成一个 Skill。Anthropic 对未来的判断也顺着这个逻辑往前推了一步。他们认为，之后 Skills 可能会有更智能的发现机制，Claude 能根据场景主动推荐相关 Skills；Skills 之间也可能互相组合，比如 deploy-to-prod 自动调用 test-runner 和 security-review；甚至还能根据仓库实际情况动态生成或更新。更远一点，还会有跨团队共享的公开市场，让常用框架的最佳实践在不同团队之间流动起来。

所以，Skills 真正值得收藏的点，不是它多像一篇新鲜指南，而是它把“团队知道但总会忘、总会重复、总会出错”的那些东西，变成了会执行、会保护、会沉淀的扩展。现在最该问的，反而不是“我能不能也做一个 Skill”，而是“我手里有没有一类重复工作，已经多到该被固化成文件夹、脚本和钩子了”。如果答案是有，那它就不是一个可选项，而是你们把 AI 编程体验再往前推一档的入口。
