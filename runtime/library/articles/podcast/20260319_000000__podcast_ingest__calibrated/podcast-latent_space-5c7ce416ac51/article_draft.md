# 先给 Claude 一台电脑，还是先给它更聪明的嘴？Anthropic押注前者

*Felix Rieseberg讲的不是把 AI 聊得更顺，而是把它塞进 VM、Chrome 和本地工作流里，让它从“会回答”变成“能交付”。*

真正刺眼的，不是 Claude Cowork 这套东西看起来更顺手，而是 Anthropic 的判断变了：AI 不该先拥有一个更聪明的聊天框，而该先拥有一台自己的电脑。Felix Rieseberg 在这场对话里不断把问题往这个方向推，他不是从“怎么把产品做得更像应用”讲起，而是直接把桌面、虚拟机、Chrome 和本地文件系统摆在台面上。

这并不意外。Felix 不是第一次在桌面软件的缝隙里下注的人。他参与过 Slack desktop app，也是在 Electron 这条路上走得最深的人之一，甚至做过把 Windows 95 跑进 Electron、再在 macOS、Windows 和 Linux 上都能启动的那种恶作剧式项目。你很难找到比他更适合讨论“AI 应不应该有自己的电脑”的人：他太清楚一层系统到底能给上层产品多少自由，也太清楚一层系统到底会拖慢什么。

Claude Cowork 的起点也带着一点意外。Anthropic 先看到的，不是“大家想用 Claude Code 写代码”，而是越来越多的人把它拿去做脏活、杂活、知识工作：报销、收据、知识库、视频上传、文件整理。连技术人自己，也开始把它用在非技术任务上。Felix 给出的定义很直接：Cowork 是 Claude Code 的用户友好版，但“用户友好”不是降级，而更像一种更大的外壳。像当年的 VS Code，看上去更轻，但最后证明自己更容易被改造、被扩展、被塞进各种工作流里。

这件事最值得注意的地方，不是产品名，而是 Anthropic 的做事方式。Felix 说 Cowork 大概 10 天就做出来了，但那不是从零开始的 10 天，而是把过去一年半里内部已经跑过的原型、组件和判断拼成了一个可以对外的东西。更关键的是，他反复强调现在的世界变了：执行变便宜了。以前要靠 memo、spec、争论和反复拉齐，现在更像是“把候选项都做出来，再拿现实选最好的那个”。这不是一句效率口号，而是产品组织方式的改变。

而 Cowork 真正的分叉点，不在界面，而在它站在哪台机器上。Felix 认为硅谷整体低估了本地电脑的价值。原因并不神秘：你的工具、文件、登录态、浏览器 cookie、工作语境，全都在本地。把 Claude 放到“你工作的地方”，往往比把所有东西搬到云端更有力量。云端当然诱人，但如果要把整台工作电脑克隆上云，很多人未必真的愿意。那已经不只是权限问题，而是工作和身份被一起搬走的问题。

所以 Cowork 选择了 VM。这个选择同时解决了两件事：一是安全边界，二是能力边界。Claude 被放进一个轻量虚拟机里，能装 Python、能装 Node.js、能跑脚本、能更独立地干活，但网络进出和文件系统又是被约束的。Felix 说得很直白：长期看，“每条命令都要人工批准”不是好体验；但“完全放飞”也不是。虚拟机提供的，恰好是中间地带。它不是把风险消灭，而是把风险关进一个更可控的盒子里。

这也解释了为什么 Cowork 和 Claude Code 的差别，并不只是“一个是桌面版，一个是终端版”。Anthropic 在它们身上看的指标本来就不一样：Claude Code 更偏编码任务，Cowork 更偏知识工作。前者看代码任务的表现，后者看财务、法务、个人资产管理这类更长、更模糊、更容易走偏的任务。Felix 透露，他们对 Cowork 的做法会更重视 planning 和 ask-user 之类的工具，因为这些任务往往不是“写得快”而是“别误解我”。你能感觉到，Cowork 的目标不是更会回答，而是更会把任务接住。

他说 eval 的时候，意思也很具体：不是随便做 vibe test，而是把完整 transcript、工具、文件输出和 token 输出一起看，衡量改动后结果怎么变。这个定义很重要，因为它说明 Anthropic 不是在调一个聊天窗口，而是在调一条工作链。很多产品把“计划”做成展示层，Cowork 更像是把计划当成工作本体的一部分。用户看到的九步计划、能编辑的中间步骤、不断 ask clarification 的过程，不是装饰，而是为了把长任务从一口气赌命，变成可以检查、可以纠偏的工作节拍。

技能是另一个更安静、但更可能改写格局的点。Felix 对 skills 的偏爱，几乎是把它当成新一层抽象：不是把能力锁在一个巨大的平台 API 里，而是把它变成文件、Markdown、文本说明。给模型一个 endpoint、一个描述、一个文件，它自己会找路。Anthropic 甚至把 skills 和插件做成了可以跨 Claude Code 和 Cowork 的同一套格式。这里最有意思的，不是“模型能不能理解说明书”，而是“说明书能不能搬家”。个人技能、工作技能、不同 agent 之间的可移植性，可能比某个单点工具做得多漂亮更重要。

swyx 在现场给了一个很能说明问题的例子：先让 Cowork 帮他把 Zoom 录音上传到 YouTube，再让它进一步把下载、命名、整理都纳入 skill，甚至自己拆分成多个 skill，再让一个父 skill 去编排。这个过程很像把一次性的人类动作变成可复用的流程，但更准确地说，它是在把“信任”一点点外包出去。先让它做小事，再让它做更大的事，最后把一个原本只能靠手工点击的流程，变成一套可以反复调用的私人基础设施。Felix 很喜欢这个类比：像玩 Factorio，只不过工厂是你自己的生活。

但他并没有把这件事浪漫化。相反，真正让他停顿的，是这套能力对劳动市场的压力，尤其是对入门级岗位的压力。Anthropic 公开承认过，他们担心这会影响市场，而且大家并没有准备好。Felix 甚至把自己大学时期的观察拿出来对照：像滑铁卢那样大量实习、在真实公司里轮转的新人，往往比纯理论训练出来的毕业生更快进入状态。问题在于，AI 让“新人阶段”本身被压缩了。它一边提高老手产出，一边抽走了本来会给新人的那部分工作。

这也是这场对话最硬的一层。Cowork 的方向，表面上是“让 AI 更像一个会干活的搭档”，实际上是在逼人回答另一个更难的问题：当 AI 真能接住你的流程、读你的文件、看你的浏览器、写你的脚本、按你的节奏推进任务时，人的角色到底是什么？Felix 没有给出一个轻飘飘的答案。他更接近一种工程师式的现实主义：先把能力做出来，再把安全、可移植性、审批疲劳和组织边界一层层补上。这个顺序并不温柔，但可能更诚实。

所以，Anthropic 说 AI 应该有自己的电脑，并不是在讲一个炫技功能，而是在回答一条产品路线：不要把 AI 限死在聊天里，也不要假装所有工作都能平滑上云。真正的分野，可能是它能不能在你的本地世界里持续工作、持续记住上下文、持续做出不需要你每一步都盯着的动作。至于这会把多少工作变成自动化，多少岗位变成更短的训练场，多少用户会愿意把整台电脑交给一个 VM，这些问题现在都还没被解决。可至少有一点已经很清楚：AI 的下一个门槛，不是更会说话，而是更值得你把电脑让出去。
