# 当“AI 黑粉”开始用 Claude 写编译器，他真正改口的不是工具，而是方法

*Steve Klabnik 从公开质疑 AI，到用 AI 重启 Rue，这个转折最值得看的地方不在立场翻盘，而在他如何把“能不能用”改成“怎么验证、怎么协作、怎么保住质量”。*

如果一个曾经自称“AI 黑粉”的人，最后把一门编程语言的大量工作交给 Claude，会发生什么？在《Practical AI》这期对话里，Steve Klabnik 给出的答案不是投降，也不是拥抱，而是一次很典型的程序员式改口：不是先问 AI 值不值得信，而是先问它到底在什么条件下开始变得有用。

Steve 的背景本来就带着这种复杂性。他做过 Rust 的长期文档工作，写过《The Rust Programming Language》，也一直对编译器、语言设计和写作之间的交界地带着迷。可到了 2025 年初，他还是会把自己归到“AI 反对者”那一边。转折并不是来自某篇技术博客，而是来自两个很不一样的现场：一个是和不在技术圈里的人聊天，另一个是他开始认真试用 agentic coding 工具。

前者让他意识到，很多争论其实连“事实”都没对齐。他提到，自己一边在用 ChatGPT 搜网页，一边又看到别人断言“它根本不能搜索网页”。这类冲突让他觉得，问题不只是观点相左，而是讨论的底层现实都没统一。后者则更直接：当他在 VS Code 里装上代理式编程工具后，他第一次感到这东西“不是玩具，而是能工作”。

真正改变他判断的，不是 AI 开始神奇，而是他开始承认：人类程序员对“电脑应该确定性地执行指令”这件事，早就形成了过度专业化的直觉。对普通人来说，电脑一直就是会“差不多对、偶尔错”的魔法盒子。Steve 说，他那位不写代码的家人会先起草、再人工核对、最后修正输出里的 10% 错误，但依然认为这些工具极大提高了写报告的效率。换句话说，非程序员并不天然更笨，反而可能更适合处理一个本来就带有噪声的系统。

这也是他这次最重的判断：软件工程师不该默认自己比普通用户更懂“工具到底该怎么用”。他直说，很多开发者习惯把 AI 的错误直接等同于“工具无用”，但这背后其实混着一种职业傲慢。对他来说，接下来更重要的不是站队，而是重新审视那些我们过去太容易当成真理的工作习惯。

Rue 就是在这个背景下重新启动的。

Steve 说，他一直想再做一次编译器项目，但自己已经没有年轻时那种把整个周末都耗在一个 bug 上的时间和精力了。Rue 第一轮在去年 8 月启动，后来因为进度卡住、模型能力和个人熟练度都不够，渐渐停了下来。到了 11 月、12 月，他重新来过，这次路线完全不同：先写语言规范，再围绕规范做验证，再让 Claude 去反复迭代实现。

这个变化很关键。对 agentic coding 来说，最值钱的不是“让模型写更多代码”，而是“让模型自己知道什么叫对”。Steve 给 Rue 做了自定义测试框架 Klab，把语言规范和测试样例连起来，让编译器输出、错误信息和程序行为都能自动验证。这样一来，模型不是靠人类一句一句手工检查，而是围着一个明确的正确性目标收敛。他把这称为一种更有效的开发方式，也承认这是自己这次做得比第一次更好的核心原因之一。

Rue 的定位也很清楚：它不是要正面替代 Rust，更不是复制 Go，而是想站在两者之间。Rust 提供极致性能和低层控制，Go 提供更顺滑的开发体验，而 Steve 想找的是一个介于两者之间的位置：比 Go 更能往底层下探一点，但又不要求像 Rust 那样把一切都押在极致性能和复杂控制上。Rue 目前不引入垃圾回收，转而借用了 Swift 和 HiLo 一类语言里“mutable value semantics”的思路，尽量通过减少引用、放弃一部分表达力，来换取没有 borrow checker 和 lifetimes 的体验。

这不是一个已经成熟的答案，反而暴露了它还很早期。他也坦白，Rue 现在还有明显的功能缺口，比如把字符串放进数组后还没法顺畅取出或访问；他甚至说自己正卡在标准库正式推进之前的门槛上。也正因如此，Rue 现在更像一个实验台，而不是一个可以直接交付普通开发者的产品。

但实验台恰好也是今天最重要的地方。

Steve 这期最值得被认真听见的，不是“AI 会不会取代程序员”这种老问题，而是他对工程实践的重排。他开始怀疑一些过去被奉为铁律的东西，比如过度强调 DRY、过分执着于一开始就把代码写得极致干净。他举了一个很直接的例子：当 Claude 提示代码里有重复函数时，他的第一反应已经不再是“这破坏了原则”，而是“那就让它先存在，后面再修”。在他看来，AI 时代的开发也许需要一点“允许先不完美”的耐心，因为很多问题其实不值得在最早阶段耗尽人力去消灭。

这种态度并不温柔，甚至带着一点危险。他自己也没有回避：让 Claude 自动合并 PR、允许更快的提交节奏，确实会显著提高速度，但质量如何维持，才是整个行业下一步真正的难题。对他而言，未来不是“要不要用 AI”的选择题，而是“如何在速度和可靠性之间重新划线”。

如果说这期对话留下了什么结论，那就是 Steve 已经不再把 AI 当成一个立场问题，而是当成一个工程问题。立场会变，工具会变，模型会变，最后不变的只有一个问题：当机器真的开始替你写、替你改、替你合并，你凭什么知道它还在朝正确的方向走？
